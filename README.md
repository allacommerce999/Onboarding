# Onboarding Agent

Агент готовит onboarding-пакет для нового сотрудника рабочей группы корпоративных систем.

> All examples are synthetic. Do not commit real employee data, internal resource names, credentials, IPs, or access routes.

Основной контекст рабочей группы:

- 1С;
- Парус;
- Oracle;
- PostgreSQL;
- MySQL;
- Microsoft SQL Server;
- Cognos.

## Что генерирует агент

1. Onboarding-план на период до выхода, первый день, первую неделю, 30/60/90 дней.
2. Список доступов с владельцами, сроками и статусами.
3. Welcome-письмо новому сотруднику.
4. Пакет заявок на доступ к ИСР, базовым ресурсам, БД, серверам, Cognos и сервисным учетным записям.
5. Пакет первичных инструктажей и локальных процедур.
6. Открытые вопросы и чеклист безопасности.

## Быстрый старт

1. Заполнить `templates/core/employee_input_form.md`.
2. Передать данные агенту вместе с `templates/access/workgroup_access_matrix.md`.
3. Попросить режим:
   - `draft` - черновик с плейсхолдерами;
   - `final` - готовый пакет после заполнения обязательных данных;
   - `synthetic` - тестовые документы без реальных ФИО и ресурсов.
4. Проверить результат по `templates/core/quality_checklist.md`.

## Обязательные входные данные

- ФИО сотрудника;
- должность;
- команда;
- дата выхода;
- формат работы;
- руководитель;
- buddy или наставник;
- HR/PeopleOps контакт;
- основные задачи роли;
- нужные системы: 1С, Парус, Oracle, PostgreSQL, MySQL, Microsoft SQL Server, Cognos;
- уровень доступа: пользователь, аналитик, разработчик, инженер сопровождения, DBA, администратор, автор отчетов.
- формат объекта: офис, удаленно, гибрид, выезд;
- нужны ли пропуск, парковка, кабинет/ключи и оборудование;
- есть ли выездные работы, электроустановки или работы на высоте.

## Инструктажи и локальные процедуры

Агент может подготовить public-safe пакет для первичного onboarding:

- матрицу обязательных инструктажей по роли и формату работы;
- короткие synthetic-примеры самих инструкций для объяснения сотруднику;
- простые чек-листы ПВТР, ОТ, пожарной безопасности, первой помощи, ЛВС/ИБ, оборудования, пропуска, ключей и парковки;
- HTML-мини-проверки знаний после инструктажей;
- отдельные блоки для электроустановок и работ на высоте со статусом `требует профильной проверки`;
- синтетические примеры и интерактивный HTML-чеклист без внешних зависимостей.

Агент не выдает юридическое заключение, не подтверждает обучение и не назначает допуски. Реальные инструкции, журналы и формулировки должны быть проверены HR, ОТ, ИБ и профильными ответственными.

## Правила безопасности

- Не указывать пароли, токены, приватные ключи и одноразовые коды.
- Не выдавать production, DBA, admin, finance и service account доступы автоматически.
- Все чувствительные доступы помечать как `требует согласования`.
- Все профильные допуски и опасные работы помечать как `требует профильной проверки`.
- Если ресурс, владелец или роль неизвестны, ставить `требует уточнения`.
- Не переносить реальные данные из `doc/` в примеры или промты.
- Не переносить реальные данные из `instr/` в примеры или промты.

## Принципы именования

| Сущность | Принцип | Пример |
|----------|---------|--------|
| Ресурс | `TEST-<SYSTEM>-<MODULE>-<ENV>` | `TEST-1C-ERP-DEV` |
| Сервер БД | `db-<system>-<env>-<nn>` | `db-parus-test-021` |
| Сервер приложений | `app-<system>-<env>-<nn>` | `app-1c-dev-106` |
| Терминальный сервер | `ts-<system>-<env>-<nn>` | `ts-cognos-test-012` |
| Пользователь | `surname + initials` | `testovds` |
| Подрядчик | `ext_ + surname + initials` | `ext_testovds` |
| Сервисная учетная запись | `svc_ + system + purpose` | `svc_parus_rpt` |
