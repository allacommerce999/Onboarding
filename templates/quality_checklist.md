# Чеклист качества onboarding-пакета

## Полнота

- [ ] Есть карточка сотрудника.
- [ ] Есть onboarding-план: до первого дня, первый день, первая неделя, 30/60/90 дней.
- [ ] Есть список доступов.
- [ ] Есть welcome-письмо.
- [ ] Есть пакет заявок на доступ, если он был запрошен.
- [ ] Есть открытые вопросы.
- [ ] Есть проверка безопасности.

## Входные данные

- [ ] Указаны ФИО, должность, команда и дата выхода.
- [ ] Указан формат работы.
- [ ] Указаны руководитель, buddy и HR/PeopleOps.
- [ ] Указаны основные системы: 1С, Парус, Oracle, PostgreSQL, MySQL, Microsoft SQL Server, Cognos.
- [ ] Неизвестные значения помечены как `требует уточнения`.

## Доступы

- [ ] У каждого доступа есть система, ресурс, контур, цель, роль, владелец, срок и статус.
- [ ] Production-доступы помечены как `требует согласования`.
- [ ] Admin/DBA-доступы помечены как `требует согласования`.
- [ ] SSH/RDP-доступы помечены как `требует согласования`.
- [ ] Сервисные учетные записи помечены как `требует согласования`.
- [ ] Для Cognos указана роль: viewer, report author, framework modeler или admin.
- [ ] Для БД указана СУБД, ресурс, схема/БД и роль.

## Безопасность

- [ ] Нет паролей.
- [ ] Нет токенов.
- [ ] Нет приватных ключей.
- [ ] Нет одноразовых кодов.
- [ ] Нет connection string с секретами.
- [ ] Нет реальных персональных данных в синтетических примерах.
- [ ] Нет реальных ресурсов из папки `doc/` в синтетических файлах.

## Welcome-письмо

- [ ] Указана дата выхода.
- [ ] Указан формат работы.
- [ ] Указан руководитель.
- [ ] Указан buddy.
- [ ] Указан HR/PeopleOps контакт.
- [ ] Тон письма теплый и деловой.
- [ ] В письме нет секретов и приватных ссылок.

## Финальная проверка

- [ ] Документ можно передать на ручное согласование.
- [ ] Все спорные доступы вынесены в открытые вопросы.
- [ ] Статусы не создают впечатление, что доступ уже выдан без подтверждения.
