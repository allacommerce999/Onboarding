# Onboarding Agent — план выполнения задачи

## 1. Цель

Создать агента, который по данным о новом сотруднике и команде генерирует пакет онбординга:

- onboarding-план по шаблону;
- список необходимых доступов;
- welcome-письмо для нового сотрудника.

Агент должен помогать HR, руководителю команды или PeopleOps быстро подготовить понятный стартовый пакет без ручного копирования шаблонов.

## 2. Роль агента

**Роль:** HR/PeopleOps onboarding assistant.

**Основная задача:** собрать входные данные, проверить полноту информации, заполнить шаблоны и выдать готовые материалы для согласования.

**Границы ответственности:**

- агент не выдает реальные пароли, токены и секреты;
- агент не назначает доступы самостоятельно, а формирует чеклист для ответственных;
- агент помечает неизвестные данные как `требует уточнения`;
- агент не придумывает внутренние системы компании, если их нет во входных данных или матрице доступов.

## 3. Входные данные

### Обязательные

- ФИО сотрудника;
- должность/роль;
- команда/отдел;
- дата выхода;
- формат работы: офис, удаленно, гибрид;
- руководитель;
- HR/PeopleOps контакт;
- наставник или buddy, если назначен;
- город/часовой пояс, если важен для рабочего графика.

### Желательные

- грейд или уровень роли;
- испытательный срок;
- основные задачи на первые 30/60/90 дней;
- стек, инструменты и системы команды;
- ссылки на внутренние регламенты;
- расписание первого дня;
- корпоративный тон общения для письма;
- матрица доступов по ролям.

### Если данных не хватает

Агент должен сначала вывести короткий список уточняющих вопросов. Если пользователь просит подготовить черновик без уточнений, агент заполняет отсутствующие поля плейсхолдерами.

## 4. Выходные материалы

Агент должен генерировать один пакет из трех документов или трех секций в одном ответе.

### 4.1. Onboarding-план

Структура:

- краткая карточка сотрудника;
- цели онбординга;
- подготовка до первого дня;
- первый день;
- первая неделя;
- первые 30 дней;
- 60 дней;
- 90 дней;
- встречи и ответственные;
- критерии успешного прохождения онбординга;
- открытые вопросы.

### 4.2. Список доступов

Формат таблицы:

| Система | Зачем нужен доступ | Уровень доступа | Ответственный | Срок | Статус |
|---------|---------------------|-----------------|---------------|------|--------|

Статусы:

- `нужно запросить`;
- `запрошено`;
- `выдано`;
- `не требуется`;
- `требует уточнения`.

Типовые категории доступов:

- корпоративная почта;
- календарь;
- мессенджер;
- таск-трекер;
- база знаний;
- репозитории;
- дизайн-система или Figma;
- CRM/ERP/BI, если применимо;
- VPN/SSO;
- рабочее оборудование;
- финансовые, production или admin-доступы только через отдельное согласование.

### 4.3. Welcome-письмо

Структура:

- тема письма;
- приветствие;
- короткое представление команды;
- дата и формат первого рабочего дня;
- что подготовить заранее;
- с кем связаться;
- дружелюбное завершение.

Тон: теплый, деловой, без чрезмерной неформальности.

## 5. Логика работы агента

1. Принять данные о сотруднике и команде.
2. Проверить обязательные поля.
3. Если критичных данных нет, задать уточняющие вопросы.
4. Нормализовать даты, имена, роли и формат работы.
5. Выбрать шаблон onboarding-плана.
6. Сформировать план по этапам: до выхода, день 1, неделя 1, 30/60/90 дней.
7. Сформировать список доступов на основе роли и матрицы систем.
8. Отделить стандартные доступы от чувствительных доступов.
9. Сгенерировать welcome-письмо.
10. Провести самопроверку результата.
11. Выдать итоговый пакет в Markdown.

## 6. Архитектура будущей реализации

```text
User Input
  -> Input Validator
  -> Employee Context Builder
  -> Template Selector
  -> Onboarding Plan Generator
  -> Access Checklist Generator
  -> Welcome Email Generator
  -> Quality Review
  -> Markdown Output
```

### Компоненты

1. **Input Validator**
   - проверяет обязательные поля;
   - формирует список уточняющих вопросов;
   - не пропускает пустые критичные значения без явной пометки.

2. **Employee Context Builder**
   - собирает единую карточку сотрудника;
   - нормализует роль, дату выхода, формат работы;
   - добавляет контекст команды.

3. **Template Selector**
   - выбирает шаблон по роли, отделу и формату работы;
   - использует базовый шаблон, если специализированного нет.

4. **Onboarding Plan Generator**
   - формирует этапы онбординга;
   - добавляет задачи, встречи и критерии успеха;
   - назначает ответственных.

5. **Access Checklist Generator**
   - подбирает доступы по роли;
   - маркирует чувствительные доступы;
   - указывает владельца и срок.

6. **Welcome Email Generator**
   - создает письмо в согласованном тоне;
   - не раскрывает внутренние секреты;
   - оставляет плейсхолдеры для ссылок и контактов, если они неизвестны.

7. **Quality Review**
   - проверяет полноту трех выходных материалов;
   - ищет противоречия в датах, именах и ответственных;
   - проверяет, что все неизвестные данные помечены.

## 7. План реализации

### Этап 1. Подготовить спецификацию

- [ ] Описать целевую роль агента.
- [ ] Зафиксировать обязательные и optional-поля.
- [ ] Согласовать формат итогового Markdown-пакета.
- [ ] Определить список ролей, для которых нужны отдельные шаблоны.

### Этап 2. Подготовить шаблоны

- [ ] Создать базовый шаблон onboarding-плана.
- [ ] Создать шаблон списка доступов.
- [ ] Создать шаблон welcome-письма.
- [ ] Добавить плейсхолдеры для неизвестных данных.

### Этап 3. Описать матрицу доступов

- [ ] Составить базовые категории систем.
- [ ] Разделить доступы на стандартные и чувствительные.
- [ ] Добавить ответственных за каждую систему.
- [ ] Добавить правила согласования для production/admin/finance-доступов.

### Этап 4. Реализовать генерацию

- [ ] Реализовать сбор входных данных.
- [ ] Реализовать проверку обязательных полей.
- [ ] Реализовать генерацию трех документов.
- [ ] Добавить режим `draft`, если пользователь хочет черновик без всех данных.

### Этап 5. Добавить проверку качества

- [ ] Проверить, что onboarding-план содержит этапы до 90 дней.
- [ ] Проверить, что список доступов содержит ответственного и статус.
- [ ] Проверить, что welcome-письмо содержит дату выхода и контактное лицо.
- [ ] Проверить отсутствие секретов, паролей и токенов.

### Этап 6. Протестировать

- [ ] Тест: сотрудник в офисе.
- [ ] Тест: удаленный сотрудник.
- [ ] Тест: не хватает обязательных данных.
- [ ] Тест: роль с чувствительными доступами.
- [ ] Тест: генерация черновика с плейсхолдерами.

## 8. Пример минимального запроса к агенту

```text
Подготовь onboarding-пакет:
- сотрудник: Тестов Иван
- роль: Backend Developer
- команда: Platform
- дата выхода: 2026-06-01
- формат: удаленно
- руководитель: Руководитель Тестовый
- buddy: Наставник Тестовый
- HR: HR Тестовая
- стек: Python, PostgreSQL, GitHub, Jira
```

## 9. Критерии приемки

- [ ] Агент выдает три обязательные части: план, доступы, письмо.
- [ ] В плане есть этапы: до первого дня, первый день, первая неделя, 30/60/90 дней.
- [ ] В списке доступов есть система, цель, уровень доступа, ответственный, срок и статус.
- [ ] Welcome-письмо можно отправить после минимальной ручной проверки.
- [ ] Неизвестные данные явно помечены как `требует уточнения`.
- [ ] Чувствительные доступы не выдаются автоматически и требуют согласования.
- [ ] В результате нет секретов, паролей, токенов и приватных ключей.

## 10. Следующие артефакты

После согласования плана можно добавить в папку:

- `README.md` — инструкция по использованию агента;
- `onboarding_template.md` — шаблон onboarding-плана;
- `access_matrix_template.md` — шаблон матрицы доступов;
- `welcome_email_template.md` — шаблон welcome-письма;
- `onboarding_agent_prompt.md` — системный промт агента;
- `examples/` — примеры входных данных и готовых результатов.

## 11. Статус реализации

Минимальный рабочий комплект агента создан:

- `README.md` — инструкция и быстрый старт;
- `onboarding_agent_prompt.md` — основной промт агента;
- `templates/employee_input_form.md` — форма входных данных;
- `templates/workgroup_access_matrix.md` — матрица доступов для рабочей группы 1С, Парус, Oracle, PostgreSQL, MySQL, Microsoft SQL Server, Cognos;
- `templates/access_request_bundle_template.md` — шаблон пакета заявок на доступ;
- `templates/quality_checklist.md` — чеклист проверки результата;
- `examples/` — синтетические примеры документов и готовых onboarding-пакетов.

Папка `doc/` используется только как локальный источник структуры и не должна попадать в git.
